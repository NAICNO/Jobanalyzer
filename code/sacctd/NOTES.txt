/*
   It looks like if we use -s to select by state we have to use a start/end time window and we will
   only see jobs that entered that state in that time window.  For example with -s COMPLETED
   we'll see jobs that completed in the time window given.

bash-5.1$ sacct -aP -s CD -S now-2hours -E now -o Account,State,User,Start,End,JobName
Account|State|User|Start|End|JobName
ec232|COMPLETED|ec-pubuduss|2024-08-01T02:18:39|2024-08-01T08:26:58|nf-germline_cpu_gatk_bqsr_(1)
ec232|COMPLETED||2024-08-01T02:18:39|2024-08-01T08:26:58|batch
ec232|COMPLETED||2024-08-01T02:18:39|2024-08-01T08:26:58|extern

   Expanding that to now-24hours would greatly expand the list.

   So if we run sacctd every hour with a a two hour window ("now-2hours" .. "now"), we should see
   jobs that entered that state in that two-hour window, and normally there would be overlap between
   runs, but the records should be identical and easily filtered (for some states).

   The states we most care about are "CANCELLED", "COMPLETED", "DEADLINE", "FAILED",
   "OUT_OF_MEMORY", and "TIMEOUT", because these pertain to how the job behaved.

   We probably should not do RUNNING unless we have good reason to.

   We can filter against the JobName: exclude "interactive" for sure, and maybe "batch" and "extern"
   (less sure about those, working on it).  I note for example a job where the JobName="batch" step
   signalled OUT_OF_MEMORY and the main step then signalled CANCELLED.  Presumably this is not
   uninteresting.

   We could filter tiny jobs (by reservation size), or not.  Might be easier to filter at the sink.

   Not obvious that this isn't just a shell script and that all the parsing should be done at the
   sink - depends on whether we do further filtering here.  Note for example that if we want to
   filter out interactive multi-step jobs we must first discover which jobs are interactive, then
   cull all other lines with the major job ID of the first line.  This requires enough parsing and
   accumulation that we might do a proper program.

   There's also an advantage to using CSV or JSON - it makes ingest easier on the JobAnalyzer side.

   Data volume is unclear and may depend greatly on whether we filter small jobs, or filter by eg
   account name (excluding sysadmin accounts for example).  The data compress exceedingly well
   however.

   $ sacct -aP -s CANCELLED,COMPLETED,DEADLINE,FAILED,OUT_OF_MEMORY,TIMEOUT -S now-2hours -E now -o JobId,JobName,User,Account,State,Start,End,ReqCPUS,ReqMem,CPUTimeRaw


   But what are the *questions* we are asking?

   - relationship between used memory and requested memory, maybe this is MaxRSS or AveRSS
   - relationship between used cores and requested cores, this is related to AveCPU
   - but we are also asking about *balance* and maybe that's where sonar data come in

   sacct will tell us what was requested - though not nearly as much as we want to know.  It doesn't
   have the script.  It doesn't know the modules.

   we get *slightly* more accurate usage info than we get from sonar, depending on the running time of
   the steps.

   we get some request information - requested cpus, requested memory.  this is the main win.

   we get the reason a job ended.

   we get account information, which may be useful for reporting to mgmt.

   we probably don't care about acct info while the job is running, so reporting only terminated
   jobs is fine.

   sonar data will tell us how the job ran - about the balance of computation on all the cores it's
   running.


   squeue also can provide a lot of data about pending jobs: queue length, time a job has been
   pending.  it can display a lot of information about resource requests, it seems even more than
   sacct.  but it can't say anything about completed jobs, and so we can't really guarantee
   that we see everything.  it becomes more of a sampler.

   the question is how to handle these data.  should they be produced by different components
   running independently and reporting independently, probably to different ingest points?  or
   should they run with more coordination and report data jointly?

   i tend to think the former: sacctd can run any time and produce a lot of data, so maybe once an
   hour.  squeued might run every five minutes.

 */

//   sacctd on two different nodes, resulting in duplicated data.
